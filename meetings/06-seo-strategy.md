# 6차 미팅: SEO 전략 수립

**일시**: 2026-04-13
**참석**: PO, PM, UX/UI 디자이너, 프로덕트 디자이너, FE 엔지니어, BE 엔지니어, 마케터
**목적**: 21개 계산기 기준 SEO 전략 수립 및 실행 로드맵

---

## 회의 내용

**PO**: 오늘은 현재 개발된 21개 계산기 기준으로 SEO 전략을 구체화하려고 해요. 우리 수익 모델이 애드센스니까, 결국 오가닉 트래픽이 전부입니다. 구글과 네이버에서 어떻게 상위 노출을 잡을 건지 전략을 세워보죠.

**마케터**: 먼저 현재 상태를 점검하자면요, 기술적 SEO 기반은 잘 갖춰져 있어요. sitemap.xml, robots.txt, JSON-LD FAQ 스키마, 페이지별 고유 title과 description, Open Graph 태그까지 있으니까요. 문제는 콘텐츠 전략이에요. 지금 가이드 텍스트가 대부분 짧은 설명 수준이거든요. 구글이 좋아하는 건 E-E-A-T — Experience, Expertise, Authoritativeness, Trustworthiness예요. 특히 금융 계산기는 YMYL(Your Money Your Life) 카테고리라서 더 까다로워요.

**PM**: YMYL이면 구글이 더 엄격하게 평가한다는 거죠?

**마케터**: 네, 맞아요. 금융 관련 페이지는 "이 정보 믿어도 되나?"를 구글이 더 세밀하게 봐요. 그래서 데이터 출처 명시, 면책 조항, 전문적인 가이드 콘텐츠가 필수예요. 다행히 면책 조항 페이지랑 데이터 출처 페이지는 이미 있어요.

**프로덕트 디자이너**: 현재 사이트 구조를 SEO 관점에서 보면, 허브 앤 스포크 모델은 잘 갖춰져 있어요. 홈 → 카테고리 → 개별 계산기, 그리고 계산기 간 CTA 연결, 관련 계산기 카드까지. 내부 링크 구조는 꽤 탄탄해요. 근데 한 가지 빠진 게, 카테고리 페이지에 설명 콘텐츠가 없어요. 지금은 그냥 계산기 카드 그리드만 있거든요.

**마케터**: 정확해요. 카테고리 페이지가 thin content예요. "/finance" 페이지에 "금융 계산기란 무엇이고, 어떤 상황에서 쓰면 좋은지" 같은 서술형 콘텐츠가 200~300자라도 있어야 해요. 구글 봇이 "이 페이지가 뭘 다루는 페이지인가"를 판단하는 데 그게 필요합니다.

**FE 엔지니어**: 기술적으로 하나 짚을 게 있는데요, 지금 우리 계산기 대부분이 "use client"로 되어 있어서, 초기 HTML에는 빈 셸만 나가요. 구글 봇이 자바스크립트를 실행해서 렌더링하긴 하지만, 크롤링 예산(crawl budget)이 제한되어 있으니까 서버 사이드에서 기본 콘텐츠라도 내려주면 유리해요.

**BE 엔지니어**: 그게 맞아요. 다만 계산기 특성상 인터랙티브한 부분은 클라이언트에서 처리해야 하니까, 정적인 부분 — h1, 가이드 텍스트, FAQ, 관련 계산기 — 이것들을 서버 컴포넌트로 빼고, 계산기 인터랙션만 클라이언트 컴포넌트로 감싸는 방식이 이상적이에요. 지금 바로 전부 리팩토링하기보다는 킬러 계산기 3개부터 적용하면 되겠어요.

**FE 엔지니어**: 동의해요. 연봉 계산기는 이미 서버/클라이언트 분리가 되어 있고, 나머지 두 개도 같은 패턴으로 가면 됩니다.

**UX/UI 디자이너**: SEO 관점에서 사용자 시그널도 중요하잖아요. 바운스율이랑 체류 시간. 지금 우리 계산기가 즉시 결과를 보여주는 건 좋은데, 결과를 본 다음에 사용자를 잡아둘 장치가 부족해요. "계산 결과를 활용하는 팁"이나 "비슷한 상황에서 알아두면 좋은 것" 같은 섹션이 있으면 체류 시간이 늘어요.

**마케터**: 그래서 제가 콘텐츠 전략을 크게 세 층으로 나눠봤어요.

첫 번째 층은 **가이드 콘텐츠 강화**. 지금 GuideText가 접혀 있는데, 그 안의 내용이 너무 짧아요. 각 계산기별로 500~1,000자 수준의 상세 가이드를 써야 해요. "연봉 실수령액 계산기"면 "2026년 4대보험 요율 변경 사항", "비과세 항목 종류", "부양가족 공제 기준" 같은 구체적인 정보를 담아야 합니다.

두 번째 층은 **블로그/콘텐츠 허브**. 계산기 자체는 도구잖아요. 근데 검색하는 사람은 "전세 vs 월세 뭐가 유리한가" 같은 질문형 검색도 많이 해요. 이런 질문에 답하는 콘텐츠를 만들고, 그 안에 우리 계산기를 임베드하는 거예요. 이건 Phase 2에서 진행할 크롤링 콘텐츠 전략과도 연결돼요.

세 번째 층은 **프로그래매틱 SEO**. 예를 들어 "연봉 5000만원 실수령액"이라는 키워드가 있어요. 이걸 "연봉 3000만원", "4000만원", "5000만원"... 이런 식으로 사전 계산된 결과 페이지를 만드는 거예요. 마치 미리 계산해놓은 것처럼.

**PM**: 프로그래매틱 SEO 얘기는 전에 나왔던 적 있는데, 페이지 수가 너무 많아지면 관리가 어렵지 않나요?

**마케터**: Next.js의 동적 라우트랑 SSG를 쓰면 되니까 관리 부담은 크지 않아요. generateStaticParams로 빌드 타임에 찍어내면 됩니다. 키워드 데이터를 보면 "연봉 4000만원 실수령액", "연봉 5000만원 실수령액" 이런 식으로 특정 금액 키워드가 개별적으로 검색량이 있어요. 이런 페이지를 30~50개만 만들어도 롱테일 트래픽이 상당해요.

**BE 엔지니어**: 기술적으로 가능해요. `/finance/salary-calculator/[amount]` 같은 동적 세그먼트를 만들고, generateStaticParams에서 주요 연봉 구간을 넣어서 빌드하면 됩니다. 각 페이지에는 해당 연봉의 계산 결과가 미리 렌더링되고, "내 연봉으로 직접 계산해보기" CTA로 메인 계산기로 연결하면 되죠.

**프로덕트 디자이너**: 주택대출도 마찬가지예요. "1억 대출 이자", "3억 대출 월 상환액" 같은 키워드가 엄청 많아요. 대출금액 × 금리 조합으로 페이지를 미리 만들 수 있어요.

**PO**: 좋은 접근인데, 한 가지 우려되는 건 구글의 "doorway page" 페널티예요. 비슷한 내용의 페이지를 대량 생산하면 오히려 역효과 날 수 있어요.

**마케터**: 맞아요. 그래서 각 프로그래매틱 페이지마다 고유한 값을 제공해야 해요. 단순히 숫자만 바꾸는 게 아니라, 해당 금액대에 맞는 맥락 콘텐츠가 있어야 해요. 예를 들어 "연봉 3000만원" 페이지에는 "사회초년생이 알아야 할 절세 팁", "연봉 8000만원" 페이지에는 "고소득자 건강보험 상한" 같은 차별화된 정보를 넣는 거죠.

**UX/UI 디자이너**: 그리고 canonical 태그 관리가 중요해요. 프로그래매틱 페이지가 메인 계산기 페이지와 중복으로 인식되면 안 되니까요.

**PM**: 정리해보면 SEO 전략이 크게 4개 축인 거죠?

**마케터**: 네. 정리할게요.

**첫째, 온페이지 SEO 강화.** 가이드 콘텐츠를 500~1,000자로 확충. JSON-LD FAQ를 3~5개로 늘림. 카테고리 페이지에 서술형 콘텐츠 추가. 이미지 alt 태그 — 지금은 이미지가 거의 없는데, 각 계산기에 대표 일러스트를 넣으면 이미지 검색 유입도 가능해요.

**둘째, 기술적 SEO 최적화.** 킬러 계산기 서버/클라이언트 분리로 초기 렌더링 개선. Core Web Vitals 관리 — 특히 Recharts 차트 때문에 LCP가 밀릴 수 있으니 lazy loading 확인. 구조화 데이터 확장 — SoftwareApplication 스키마나 WebApplication 스키마 추가.

**셋째, 프로그래매틱 SEO.** 연봉 구간별, 대출 금액별, 복리 수익률별 사전 계산 결과 페이지 생성. 중복 콘텐츠 방지를 위해 구간별 맥락 콘텐츠 차별화.

**넷째, 오프페이지 + 네이버 전략.** 네이버 서치어드바이저 등록은 당연하고, 네이버 블로그 포스팅으로 백링크 확보. 네이버 검색은 구글이랑 알고리즘이 다르거든요. 네이버는 자사 플랫폼 콘텐츠를 더 우대하니까, 네이버 블로그에 "연봉 실수령액 계산 방법" 같은 포스팅을 올리고 우리 사이트로 링크를 거는 거예요.

**FE 엔지니어**: 추가로, 사이트 속도 관련해서 몇 가지 더 할 수 있는 게 있어요. 지금 Pretendard 폰트가 CSS에서 font-family로만 지정되어 있는데, 실제로 웹폰트 파일을 호스팅하거나 CDN에서 로딩하는 설정이 안 되어 있어요. 사용자 기기에 설치 안 되어 있으면 fallback 폰트로 넘어가요. 폰트 로딩을 명시적으로 설정하면 CLS 방지에도 도움되고요.

**BE 엔지니어**: 그리고 sitemap.xml에 프로그래매틱 SEO 페이지들도 동적으로 추가해야 해요. generateStaticParams에서 생성하는 경로들을 sitemap에도 반영하는 로직이 필요합니다.

**PO**: 알겠어요. 그러면 우선순위를 정해보죠. 지금 바로 할 수 있는 것과 나중에 할 것.

**PM**: 제가 나눠볼게요.

**즉시 실행 가능한 것들**: 가이드 콘텐츠 강화, 카테고리 페이지 콘텐츠 추가, JSON-LD FAQ 확장, Google Search Console + 네이버 서치어드바이저 등록, Google Analytics 연동. 이건 코드 변경 최소화로 콘텐츠만 추가하면 돼요.

**1~2주 내 실행**: 킬러 계산기 서버/클라이언트 분리 리팩토링, Pretendard 웹폰트 설정, Core Web Vitals 점검.

**3~4주 내 실행**: 프로그래매틱 SEO 페이지 생성 (연봉 구간 30개, 대출 금액 20개 정도로 시작), 네이버 블로그 콘텐츠 5~10개.

**마케터**: 추가로, 런칭 초기에는 Google Search Console에서 URL 인덱싱을 수동으로 요청하는 게 중요해요. 사이트맵 제출하고, 주요 페이지 21개를 하나씩 인덱싱 요청하면 크롤링 속도가 빨라져요.

**PO**: 네이버 쪽은?

**마케터**: 네이버 서치어드바이저에서 사이트 등록하고, 마찬가지로 사이트맵 제출해요. 그리고 네이버 쪽 키워드 전략이 좀 달라요. 네이버는 "연봉 실수령액 계산기"보다 "연봉 실수령액 계산"이나 "연봉 세후 계산" 같은 좀 더 자연어적인 키워드가 잘 먹혀요. title 태그를 양쪽 플랫폼에 최적화할 순 없으니까, description이나 가이드 콘텐츠에 네이버향 키워드를 자연스럽게 녹여야 합니다.

**프로덕트 디자이너**: 하나 더 — Open Graph 이미지요. 지금 OG 이미지가 설정 안 되어 있어서, 소셜 공유 시 미리보기가 기본 이미지로 나올 거예요. 각 계산기별로 간단한 OG 이미지를 생성하면 CTR(클릭률)이 확실히 올라요. Next.js의 og 이미지 생성 기능을 쓰면 동적으로 만들 수 있어요.

**FE 엔지니어**: `opengraph-image.tsx` 파일로 라우트별 OG 이미지를 자동 생성할 수 있어요. Vercel의 @vercel/og 패키지를 쓰면 됩니다.

**PO**: 좋아요. 오늘 나온 내용을 정리하면 4개 축 전략에 3단계 실행 로드맵이네요. 이걸로 확정합시다.

## 결론

### SEO 4축 전략

| 축 | 핵심 액션 | 기대 효과 |
|---|---|---|
| **온페이지 SEO** | 가이드 콘텐츠 500~1,000자 확충, JSON-LD FAQ 3~5개, 카테고리 페이지 설명 추가 | 페이지 품질 점수 향상, E-E-A-T 충족 |
| **기술적 SEO** | 서버/클라이언트 분리, 웹폰트 최적화, Core Web Vitals, OG 이미지 자동 생성 | 크롤링 효율성, 사이트 속도, 소셜 CTR |
| **프로그래매틱 SEO** | 연봉 구간·대출 금액별 사전 계산 결과 페이지 (50~100개) | 롱테일 키워드 대량 포착 |
| **오프페이지 + 네이버** | Search Console/서치어드바이저 등록, 네이버 블로그 백링크, 네이버향 키워드 최적화 | 인덱싱 속도, 네이버 검색 노출 |

### 실행 로드맵

| 단계 | 시기 | 작업 |
|---|---|---|
| **Phase A** (즉시) | 이번 주 | 가이드 콘텐츠 확충, 카테고리 설명 추가, JSON-LD 확장, Search Console 등록, GA 연동 |
| **Phase B** (1~2주) | 다음 주 | 킬러 계산기 SSR 리팩토링, Pretendard 폰트 설정, CWV 점검, OG 이미지 생성 |
| **Phase C** (3~4주) | 3주 차 | 프로그래매틱 SEO 페이지 50개, 네이버 블로그 포스팅 5~10개 |

### 키워드 타겟 우선순위

| 계산기 | 주력 키워드 | 월 검색량 (추정) |
|---|---|---|
| 연봉 실수령액 | "연봉 실수령액", "연봉 4000만원 실수령" | 10만+ |
| 주택대출 | "대출 이자 계산", "주담대 계산기" | 5만+ |
| BMI | "BMI 계산", "비만도 측정" | 10만+ |
| 만 나이 | "만 나이 계산", "나이 계산기" | 8만+ |
| 퍼센트 | "퍼센트 계산" | 5만+ |
| 전기요금 | "전기요금 계산", "전기세 누진세" | 3만+ |

## 다음 안건

- Phase A 즉시 실행 (가이드 콘텐츠 작성, Search Console 등록)
- OG 이미지 생성 기술 검토
- 프로그래매틱 SEO 페이지 구조 설계
