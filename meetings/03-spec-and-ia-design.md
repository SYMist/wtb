## 안건 1: 킬러 계산기 3종 상세 스펙

**PM**: 오늘은 세 가지 안건을 모두 마무리합니다. 첫째, 킬러 계산기 3종의 상세 스펙. 둘째, 일반 계산기 목록 확정. 셋째, 사이트 전체 IA와 계산기 간 흐름 설계입니다. 양이 많으니 킬러 계산기부터 바로 들어갑시다.

**PM**: 지난 미팅에서 킬러 계산기 후보로 주택대출 시뮬레이터, 연봉 실수령액 분석기, 투자 복리 계산기를 선정했습니다. 각각의 입력 항목, 출력 항목, 시각화 요소, 연결 흐름을 구체적으로 정의합시다. 주택대출부터 가시죠.

**프로덕트 디자이너**: 주택대출 종합 시뮬레이터의 핵심은 "비교"입니다. 단일 조건 계산이 아니라 여러 상환 방식을 한눈에 비교하는 게 핵심 가치예요. 입력 항목은 매매가, 자기자금 비율 또는 대출금액 직접 입력, 연이자율, 대출기간, 거치기간으로 봅니다.

**FE 엔지니어**: 연이자율은 직접 입력 외에 현재 시중 주요 은행 금리를 프리셋으로 제공하면 좋겠어요. "시중 평균 금리 적용" 버튼 하나면 사용자가 금리를 일일이 찾을 필요가 없어집니다.

**BE 엔지니어**: 금리 프리셋은 주기적으로 갱신해야 하는데, 한국은행 기준금리 API는 공개되어 있고요, 시중 은행별 금리는 금융감독원 금융상품 비교 공시 데이터에서 가져올 수 있습니다. 일 1회 빌드 시 갱신하는 정도면 충분합니다.

**프로덕트 디자이너**: 출력은 이렇게 구성하겠습니다. 상단에 핵심 요약으로 월 상환액, 총 상환액, 총 이자를 크게 보여주고요. 그 아래에 원리금균등, 원금균등, 만기일시 세 가지 방식의 비교 테이블을 놓습니다. 그 밑에 월별 상환 스케줄 차트를 두는데, 원금 부분과 이자 부분을 색으로 구분하는 스택 바 차트가 적합합니다.

**UX/UI 디자이너**: 거기에 "연봉 대비 상환 부담률" 섹션을 추가하고 싶어요. 연봉을 입력하면 DSR이 자동으로 계산되는 거예요. 그리고 이 부분이 연봉 실수령액 계산기로 넘어가는 연결 포인트가 됩니다. "정확한 실수령액이 궁금하다면?" 같은 CTA를 두면 자연스럽습니다.

**마케터**: SEO 관점에서 하나 추가하겠습니다. 이 페이지에 "2026년 주택담보대출 금리 현황" 같은 텍스트 콘텐츠 섹션을 하단에 넣으면 정보성 검색 키워드도 같이 잡을 수 있어요. 도구 페이지인데 콘텐츠 페이지 역할도 겸하는 겁니다.

**PO**: 좋습니다. 다음, 연봉 실수령액 분석기 가시죠.

**프로덕트 디자이너**: 연봉 실수령액은 사실 계산 자체는 단순해요. 연봉 넣으면 국민연금, 건강보험, 장기요양보험, 고용보험, 소득세, 지방소득세 떼고 나오는 거니까요. 차별화 포인트는 두 가지입니다. 첫째, 공제 항목별 비율을 파이 차트로 시각화해서 "내 돈이 어디로 가는지" 한눈에 보여주는 것. 둘째, 연봉 구간별 실수령액 비교 그래프를 제공하는 겁니다. "연봉이 500만 원 오르면 실수령은 얼마나 오르나?"를 시각적으로 보여주면 체류시간이 크게 늘어요.

**FE 엔지니어**: 입력 항목을 정리하면 연봉 또는 월급, 비과세액, 부양가족 수, 20세 이하 자녀 수 정도면 됩니다. 4대보험 요율과 소득세 간이세액표는 매년 바뀌는데, 이건 연초에 한 번만 업데이트하면 되니까 정적 데이터로 가져가도 무방합니다.

**UX/UI 디자이너**: 여기서 중요한 UX 포인트가 있어요. 연봉 입력에 슬라이더를 넣는 겁니다. 슬라이더를 움직이면 모든 공제 항목과 실수령액이 실시간으로 바뀌고, 차트도 동시에 갱신되는 거예요. 이게 기존 사이트와의 결정적 차이입니다. 기존 사이트는 숫자 입력하고 버튼 누르고 결과 기다리는 3단계인데, 우리는 슬라이더 하나로 끝입니다.

**프로덕트 디자이너**: 연결 흐름은 이렇습니다. 주택대출 시뮬레이터에서 연봉을 입력했으면 그 값이 연봉 실수령액 분석기로 넘어오고요. 반대로 연봉 실수령액 분석기에서 "이 연봉으로 대출 가능한 금액이 궁금하다면?" 같은 CTA로 주택대출 쪽으로 연결합니다. 양방향 흐름이에요.

**PO**: 마지막, 투자 복리 계산기요.

**프로덕트 디자이너**: 투자 복리 계산기는 두 가지 모드를 제공합니다. 첫째는 "내가 얼마를 모을 수 있나" 정방향 계산. 초기 투자금, 월 적립액, 예상 수익률, 투자 기간을 넣으면 최종 금액이 나옵니다. 둘째는 "목표 금액을 모으려면?" 역방향 계산. 목표 금액과 기간을 넣으면 필요한 월 적립액이나 수익률을 알려주는 거예요. 이 역산 기능이 기존 사이트에는 거의 없습니다.

**FE 엔지니어**: 시각화는 라인 차트가 핵심이겠네요. 원금 누적선과 수익 포함 누적선을 같이 그려서 복리 효과가 시간이 갈수록 벌어지는 걸 시각적으로 보여주는 거예요. 여기에 인플레이션 적용 선을 하나 더 추가하면 "실질 수익"을 보여줄 수 있어서 차별화가 됩니다.

**마케터**: 이 계산기는 적립식과 거치식 비교가 가능하면 좋겠어요. "매월 50만 원 적립 vs 목돈 3천만 원 거치" 같은 비교를 한 화면에서 보여주면 검색 키워드도 "적립식 거치식 비교"까지 잡을 수 있습니다.

**UX/UI 디자이너**: 연결 흐름은 연봉 실수령액 분석기에서 자연스럽게 넘어옵니다. "실수령액에서 매월 투자 가능한 금액은?" 이런 연결이요. 실수령액에서 생활비를 빼면 투자 가능 금액이 되니까요. 사용자가 연봉 입력 → 실수령액 확인 → 투자 시뮬레이션까지 하나의 여정으로 이어지는 겁니다.

**PM**: 킬러 계산기 3종 스펙이 정리된 것 같습니다. 다음 안건으로 넘어가겠습니다.

## 안건 2: 일반 계산기 목록 확정

**PM**: 템플릿 기반으로 빠르게 만들 일반 계산기 15~20종을 확정합시다. 마케터님이 검색 볼륨 기준으로 후보를 준비해오셨죠?

**마케터**: 네. 카테고리별로 정리했습니다. 검색 볼륨과 경쟁 강도를 고려해서 우선순위를 매겼어요.

금융/세금 카테고리에서는 예적금 이자 계산기, 전월세 전환 계산기, 양도소득세 계산기, 부가세 계산기, 퇴직금 계산기를 넣고 싶습니다.

건강 카테고리에서는 BMI 계산기, 기초대사량 계산기, 칼로리 계산기를 제안합니다.

단위 변환 카테고리에서는 길이/무게/부피 변환기, 평수 제곱미터 변환기, 환율 변환기를 넣겠습니다.

날짜/시간 카테고리에서는 디데이 계산기, 나이 계산기, 날짜 차이 계산기를 제안합니다.

일상생활 카테고리에서는 전기요금 계산기, 자동차 연비 계산기, 택배 배송비 계산기를 넣겠습니다.

총 18종입니다.

**프로덕트 디자이너**: 목록은 괜찮은데, 킬러 계산기와의 연결성을 기준으로 우선순위를 다시 볼 필요가 있어요. 예적금 이자 계산기는 투자 복리 계산기와 연결되고, 전월세 전환 계산기는 주택대출 시뮬레이터와 연결되고, 퇴직금 계산기는 연봉 분석기와 연결되잖아요. 이런 연결성 높은 것들을 먼저 만들어야 계산기 간 흐름이 빨리 완성됩니다.

**PO**: 맞습니다. 그러면 1차 배치와 2차 배치를 나눠야겠네요.

**프로덕트 디자이너**: 1차 배치는 킬러 계산기와 직접 연결되는 것들 위주로 하겠습니다. 예적금 이자 계산기, 전월세 전환 계산기, 퇴직금 계산기, 양도소득세 계산기, 부가세 계산기, 평수 제곱미터 변환기, 환율 변환기. 이 7종을 킬러 3종과 함께 1차로 내면 총 10개 페이지입니다.

2차 배치는 나머지 11종을 한 번에 올립니다. 템플릿이 이미 있으니까 빠르게 찍어낼 수 있어요.

**FE 엔지니어**: 템플릿 설계 관점에서 말씀드리면, 일반 계산기는 구조가 거의 동일합니다. 입력 폼 영역, 결과 표시 영역, 관련 계산기 링크 영역, 하단 SEO 콘텐츠 영역. 이 4블록 레이아웃을 컴포넌트로 만들어두면 새 계산기 추가는 설정 파일 하나 작성하는 수준으로 줄어듭니다.

**BE 엔지니어**: 일반 계산기는 전부 클라이언트 사이드 연산이니까 백엔드가 필요 없어요. 환율 변환기만 외부 API 연동이 필요한데, 이것도 빌드 타임에 가져와서 정적으로 넣으면 됩니다.

**마케터**: SEO 관점에서 하나 추가요. 각 계산기 페이지 하단에 해당 주제에 대한 가이드 텍스트를 300~500자 정도 넣어야 합니다. "양도소득세란?", "BMI란?" 같은 기본 설명이요. 이게 있어야 검색 엔진이 해당 페이지를 전문 콘텐츠로 인식합니다. 없으면 그냥 인터랙티브 위젯으로 취급되어서 순위가 안 올라요.

**UX/UI 디자이너**: 그 텍스트 영역은 접힌 상태로 제공하면 좋겠어요. 사용자 대부분은 계산이 목적이니까 기본 뷰는 깔끔하게 유지하고, 설명이 필요한 사람만 펼쳐보는 구조요. 하지만 HTML에는 포함되어 있으니 검색 엔진은 읽을 수 있습니다.

**PM**: 좋습니다. 목록과 배치 순서 확정하겠습니다.

## 안건 3: 사이트 IA 및 계산기 간 흐름 설계

**PM**: 마지막 안건입니다. 전체 사이트의 정보 구조와 계산기 간 흐름을 설계합시다.

**프로덕트 디자이너**: 사이트 IA를 제안하겠습니다. 최상위에 홈페이지가 있고, 그 아래 카테고리 페이지들이 있습니다. 금융, 건강, 단위변환, 날짜, 생활 이렇게 5개 카테고리요. 각 카테고리 아래에 개별 계산기 페이지가 위치합니다.

URL 구조는 이렇게 가겠습니다. 홈은 루트, 카테고리 페이지는 `/finance`, `/health` 같은 형태, 개별 계산기는 `/finance/loan-calculator`, `/health/bmi-calculator` 같은 형태입니다.

**마케터**: URL 구조에 대해 한 가지 의견이 있어요. SEO 관점에서는 한글 URL이 유리할 수 있습니다. `/금융/대출이자계산기` 같은 형태요. 구글은 한글 URL도 잘 인식하고, 검색 결과에서 한글 URL이 CTR이 더 높아요. 하지만 기술적 이슈가 있을 수 있으니 엔지니어 의견을 듣고 싶습니다.

**FE 엔지니어**: 한글 URL은 인코딩 문제가 있어요. 공유할 때 `%EC%9E%A5%` 이런 식으로 깨져 보이는 경우가 많고, 일부 시스템에서 호환성 문제가 있습니다. 영문 slug를 쓰되, canonical 태그와 메타 데이터를 한글로 확실하게 잡는 게 낫습니다.

**마케터**: 그러면 영문 slug로 가되, 페이지 title과 h1은 반드시 한글 키워드를 포함하는 것으로 합시다. "대출이자 계산기 - 원리금균등 원금균등 비교" 이런 식으로요.

**프로덕트 디자이너**: 동의합니다. 그러면 URL 구조는 영문으로 확정합니다. 다음으로 계산기 간 흐름을 설계하겠습니다. 핵심은 킬러 계산기 3종을 중심으로 일반 계산기들이 위성처럼 연결되는 허브 앤 스포크 구조입니다.

주택대출 시뮬레이터를 중심으로 전월세 전환 계산기, 양도소득세 계산기, 평수 변환기가 연결됩니다. 연봉 실수령액 분석기를 중심으로 퇴직금 계산기, 부가세 계산기가 연결되고요. 투자 복리 계산기를 중심으로 예적금 이자 계산기, 환율 변환기가 연결됩니다.

그리고 킬러 계산기 3종 사이에도 상호 연결이 있어요. 주택대출에서 연봉 분석기로, 연봉 분석기에서 투자 계산기로, 투자 계산기에서 다시 주택대출로. 삼각 순환 구조입니다.

**UX/UI 디자이너**: 연결 방식에 대해 구체적으로 말씀드리겠습니다. 두 가지 형태로 나눕니다. 첫째는 컨텍스트 CTA예요. 계산 결과 바로 옆에 맥락에 맞는 연결을 넣는 겁니다. 예를 들어 대출 상환액 계산 결과 아래에 "이 상환액이 연봉 대비 몇 %인지 확인하기"라는 버튼이 뜨는 거죠. 이때 현재 입력값이 다음 계산기로 전달됩니다.

둘째는 관련 계산기 섹션이에요. 각 계산기 페이지 하단에 "함께 많이 사용하는 계산기" 영역을 두는 겁니다. 이건 내부 링크 구조를 강화해서 SEO에도 도움이 되고, 사용자의 추가 탐색을 유도합니다.

**FE 엔지니어**: 데이터 전달 방식을 정리하면요. 컨텍스트 CTA를 통한 이동은 URL 쿼리 파라미터로 전달합니다. `/finance/salary-calculator?annual=50000000` 이런 식이요. 이렇게 하면 공유도 가능해져서 "내 조건으로 계산해봐"라고 링크를 보낼 수 있게 됩니다. 로컬 스토리지는 보조 수단으로만 쓰겠습니다.

**프로덕트 디자이너**: 홈페이지 구조도 정의하겠습니다. 상단에 검색바를 크게 놓고, 그 아래 킬러 계산기 3종을 카드 형태로 강조 배치합니다. 그 밑에 카테고리별로 일반 계산기를 그리드로 나열하고요. 최하단에 최근 업데이트된 데이터 정보, 예를 들어 "현재 기준금리 3.5%, 최종 업데이트 2026-04-09" 같은 신뢰도 요소를 넣겠습니다.

**PO**: 네비게이션은 어떻게 구성하나요?

**UX/UI 디자이너**: 상단 GNB에 카테고리 5개를 넣고, 각 카테고리에 호버하면 드롭다운으로 해당 카테고리의 계산기 목록이 나오는 구조입니다. 모바일에서는 햄버거 메뉴로 전환되고요. 그리고 모든 페이지 상단에 검색 기능을 유지합니다. 사용자가 "전세"만 입력해도 관련 계산기가 자동완성으로 나오게요.

**마케터**: 사이트맵과 robots.txt도 초기부터 제대로 설정해야 합니다. 그리고 각 계산기 페이지마다 구조화 데이터를 JSON-LD로 넣어야 해요. FAQPage 스키마나 HowTo 스키마를 적용하면 검색 결과에서 리치 스니펫이 노출되어 CTR이 크게 올라갑니다.

**FE 엔지니어**: Next.js에서 구조화 데이터 삽입은 간단합니다. 계산기 설정 파일에 FAQ 데이터를 포함시키면 빌드 시 자동으로 JSON-LD가 생성되게 만들겠습니다.

**BE 엔지니어**: 배포 및 인프라 구조도 확정하면 좋겠는데요. Vercel에 Next.js SSG로 배포하면 무료 플랜으로도 충분하고, 커스텀 도메인 연결도 바로 됩니다. CDN도 자동이고요. Phase 2에서 크롤링 파이프라인이 들어오면 GitHub Actions나 Vercel Cron으로 주기적 빌드를 트리거하면 됩니다.

**FE 엔지니어**: Next.js 14 App Router, TypeScript, Tailwind CSS를 베이스로 가겠습니다. 차트 라이브러리는 Recharts가 가볍고 React 친화적이라 적합합니다. 상태 관리는 계산기 정도면 React 기본 useState로 충분하고, 별도 라이브러리는 필요 없어요.
