## 안건 1: 사이트 네이밍 및 도메인 확정

**PO**: 네이밍은 두 가지 조건을 충족해야 합니다. 첫째, 계산기에 국한되지 않는 포괄적인 이름이어야 해요. Phase 2에서 크롤링 콘텐츠, 텍스트 도구 등으로 확장할 거니까요. 둘째, 한국어 사용자가 기억하기 쉬워야 합니다. 영문이어도 직관적이어야 해요.

**마케터**: 네이밍 후보를 준비해왔습니다. 다섯 가지예요.

첫째, **CalcuTools**. 계산기와 도구를 합친 이름인데, 계산기에 너무 묶이는 느낌이 있어요. 확장성이 떨어집니다.

둘째, **ToolPick**. 도구를 골라 쓴다는 의미인데, 도구 모음이라는 정체성은 잘 드러나요. 다만 좀 평범합니다.

셋째, **Tooly**. 짧고 친근한 느낌이에요. 도메인도 확보하기 비교적 쉬울 겁니다. 다만 장난감 같은 인상을 줄 수 있어요.

넷째, **SimplyCalc**. "간단하게 계산한다"는 메시지가 명확한데, 역시 계산기에 한정되는 느낌입니다.

다섯째, **AllCalc** 또는 **CalcAll**. "모든 계산"이라는 의미로 범위가 넓어요. 짧고 기억하기 쉽습니다.

**UX/UI 디자이너**: 브랜드 관점에서 보면 Tooly가 가장 캐릭터가 살아요. 로고 디자인이나 일러스트레이션에서 활용하기 좋고, 친근한 톤이 유틸리티 사이트의 접근성을 높여줍니다. 장난감 같다는 건 비주얼로 보정할 수 있어요. 깔끔한 디자인 시스템을 적용하면 오히려 "친근하지만 신뢰감 있는" 포지션이 가능합니다.

**프로덕트 디자이너**: 저는 확장성 관점에서 봅니다. CalcuTools나 SimplyCalc은 이름 자체가 "계산기 사이트"를 선언하는 거라 나중에 텍스트 변환 도구나 정보 콘텐츠가 들어오면 어색해져요. Tooly나 ToolPick이 확장에 유리합니다. 그 중에서 Tooly가 더 짧고 브랜딩 잠재력이 높다고 봅니다.

**FE 엔지니어**: 도메인 가용성 관점에서 말씀드리면, tooly.com은 이미 선점되어 있을 가능성이 높아요. tooly.kr이나 tooly.co.kr을 확인해봐야 하고, 아니면 gettooly.com, tooly.tools 같은 변형도 고려해야 합니다.

**마케터**: .kr 도메인이 한국 타겟 SEO에서 유리합니다. 구글이 지역 타겟팅 신호로 ccTLD를 인식하거든요. tooly.kr을 우선 확인하고, 안 되면 tooly.co.kr로 가는 게 좋겠어요.

**PO**: 잠깐, 한 가지 더 고려할 게 있어요. 우리 사이트가 결국 "한국어 콘텐츠, 한국 사용자 타겟"이잖아요. 영문 이름이 꼭 필요한가요? 한글 브랜드명은 어떻습니까?

**마케터**: 한글 브랜드명은 기억하기 쉬운 장점이 있지만 도메인에서 문제가 됩니다. 한글 도메인은 실질적으로 사용하기 어렵고, 결국 영문 도메인이 필요해요. 브랜드명은 영문으로 가되 한글 서브 타이틀을 붙이는 게 낫습니다. 예를 들어 "Tooly - 생활 도구 모음" 이런 식이요.

**BE 엔지니어**: 하나 추가하면, 프로젝트 리포지토리명이나 패키지명도 이름과 통일하는 게 관리하기 편합니다. 너무 긴 이름이나 특수문자가 있으면 불편하니까 짧고 알파벳만 쓰는 이름이 기술적으로도 좋습니다.

**PM**: 정리하면 Tooly가 가장 많은 지지를 받고 있네요. 이의 있으신 분?

**PO**: 없습니다. Tooly로 가되, 도메인 확보 여부에 따라 최종 확정합시다. 1순위 tooly.kr, 2순위 tooly.co.kr, 3순위 gettooly.com으로 하겠습니다.

**마케터**: 그리고 서브 타이틀은 "생활 계산기 & 도구 모음"으로 가겠습니다. SEO 타이틀에서는 이 서브 타이틀이 계속 노출되어야 해요. "Tooly - 생활 계산기 & 도구 모음"이 홈 페이지 title이 됩니다.

## 안건 2: 디자인 시스템 및 와이어프레임

**PM**: 디자인 시스템과 주요 페이지 와이어프레임을 정의합시다.

**UX/UI 디자이너**: 디자인 시스템부터 가겠습니다. 핵심 원칙은 세 가지예요.

첫째, **신뢰감 있는 깔끔함**. 유틸리티 사이트는 "이 결과를 믿어도 되나?"가 중요합니다. 과한 장식 없이 정보 위계가 명확한 디자인이어야 해요. 데이터와 숫자가 주인공이니까 타이포그래피가 특히 중요합니다.

둘째, **접근성 우선**. 우리 사용자는 20대부터 60대까지 폭이 넓어요. 대출 계산기를 쓰는 40~50대도 편하게 사용할 수 있어야 합니다. 폰트 크기 최소 16px, 터치 영역 최소 44px, 색상 대비 4.5:1 이상을 기본으로 잡겠습니다.

셋째, **광고 공존 설계**. 애드센스가 수익원이니까 광고 영역을 디자인 단계부터 고려해야 합니다. 광고가 들어가도 UX가 무너지지 않는 레이아웃을 설계해야 해요.

**프로덕트 디자이너**: 광고 배치에 대해 구체적으로 제안하겠습니다. 계산기 페이지 기준으로 광고 슬롯은 세 곳으로 잡겠습니다. 첫째, 페이지 상단 배너. 가로형 728x90 또는 반응형. 둘째, 계산 결과와 가이드 텍스트 사이. 사용자가 계산을 완료한 시점에 노출되니까 자연스러워요. 셋째, 사이드바. 데스크톱에서만 노출하고, 모바일에서는 숨깁니다.

**UX/UI 디자이너**: 중요한 원칙 하나 추가합니다. 입력 폼과 결과 출력 사이에는 절대 광고를 넣지 않습니다. 사용자의 핵심 태스크 흐름을 끊으면 이탈률이 급등해요. 광고는 태스크 완료 후에만 배치합니다.

**PO**: 동의합니다. 단기 수익을 위해 UX를 해치면 장기적으로 재방문률이 떨어지니까요.

**UX/UI 디자이너**: 컬러 시스템을 정의하겠습니다. 프라이머리 컬러는 블루 계열로 가겠습니다. 금융, 계산, 신뢰를 연상시키는 색이에요. 구체적으로 메인 블루는 `#2563EB`, 진한 변형은 `#1D4ED8`, 연한 변형은 `#DBEAFE`입니다. 세컨더리는 그린 `#16A34A`로 긍정적 결과나 이득을 표시할 때 쓰고, 레드 `#DC2626`은 부정적 결과나 손실 표시에 씁니다. 배경은 화이트 `#FFFFFF`과 라이트 그레이 `#F9FAFB` 투톤이고, 텍스트는 `#111827`이 기본, `#6B7280`이 보조입니다.

**FE 엔지니어**: Tailwind CSS 기본 팔레트와 거의 일치하네요. 커스텀 설정 없이 Tailwind 기본 클래스로 대부분 커버됩니다. blue-600, blue-700, blue-100 이런 식으로요. 개발 속도에 유리합니다.

**UX/UI 디자이너**: 의도한 겁니다. 디자인 시스템이 기술 스택과 바로 매핑되어야 구현 과정에서 디자인이 왜곡되지 않아요.

타이포그래피는 시스템 폰트 스택을 씁니다. Pretendard를 1순위로 두고, 없으면 Apple SD Gothic Neo, Malgun Gothic 순으로 폴백합니다. 숫자 표시용으로는 Tabular Figures 속성을 적용해서 숫자가 정렬될 때 자릿수가 밀리지 않게 합니다. 이게 계산기 사이트에서는 굉장히 중요해요.

**프로덕트 디자이너**: 다음으로 주요 페이지 와이어프레임을 정의하겠습니다. 먼저 **홈페이지** 구조입니다.

최상단에 GNB가 있고, 로고는 좌측, 카테고리 네비게이션이 중앙, 검색 아이콘이 우측입니다. GNB 바로 아래에 히어로 영역을 두는데, 큰 검색바를 중앙에 배치하고 "무엇을 계산할까요?"라는 플레이스홀더를 넣습니다. 그 아래 킬러 계산기 3종을 3열 카드로 배치하는데, 각 카드에는 계산기 이름, 한 줄 설명, 대표 아이콘이 들어갑니다. 그 밑에 카테고리별 섹션이 탭 형태로 나오고, 각 카테고리 안에 계산기들이 리스트 또는 그리드로 나열됩니다. 최하단 푸터에는 사이트 소개, 개인정보처리방침, 데이터 출처 고지가 들어갑니다.

**UX/UI 디자이너**: 다음으로 **킬러 계산기 페이지** 와이어프레임입니다.

좌측 또는 상단에 입력 폼 영역이 위치합니다. 데스크톱에서는 좌우 분할, 모바일에서는 상하 배치예요. 입력 항목은 그룹별로 나누고, 각 그룹에 소제목을 붙입니다. 슬라이더가 있는 항목은 슬라이더와 숫자 직접 입력을 동시에 제공합니다.

우측 또는 하단이 결과 영역인데요. 최상단에 핵심 숫자를 크게 보여주는 요약 카드가 있고, 그 아래에 비교 차트, 그 아래에 상세 테이블이 위치합니다. 결과 영역 하단에 컨텍스트 CTA 버튼이 있어서 관련 계산기로 이동할 수 있고요. 그 밑에 광고 슬롯, 그 밑에 접힌 상태의 가이드 텍스트, 마지막으로 관련 계산기 카드 3~4개가 나열됩니다.

**FE 엔지니어**: 데스크톱 좌우 분할 레이아웃에서 한 가지 제안이 있습니다. 결과 영역을 sticky로 만들면 좋겠어요. 입력 항목이 많아서 스크롤이 생기더라도 결과가 항상 보이면 실시간 갱신의 효과가 극대화됩니다. 사용자가 값을 바꿀 때마다 결과가 바로 옆에서 바뀌는 걸 볼 수 있으니까요.

**UX/UI 디자이너**: 훌륭한 제안이에요. 채택합니다. 결과 영역을 sticky로 고정하되 화면 높이를 넘어가면 내부 스크롤이 생기는 구조로 가겠습니다.

**프로덕트 디자이너**: **일반 계산기 페이지**는 킬러 계산기보다 단순합니다. 단일 컬럼 레이아웃으로 가겠습니다. 상단에 입력 폼, 중단에 결과 표시, 하단에 가이드 텍스트와 관련 계산기. 데스크톱에서는 사이드바에 광고를 두고, 모바일에서는 결과 아래에 인라인 광고를 배치합니다.

**마케터**: 와이어프레임에 한 가지 추가 요청이 있어요. 모든 계산기 페이지에 "결과 공유하기" 버튼을 넣어주세요. URL에 파라미터가 포함되니까 링크만 복사해도 동일한 계산 결과를 볼 수 있잖아요. 카카오톡 공유, 링크 복사 두 가지면 충분합니다. 이게 바이럴 유입을 만들어줘요.

**FE 엔지니어**: 카카오 공유는 카카오 SDK 연동이 필요하긴 한데 어렵지 않습니다. 링크 복사는 Clipboard API로 바로 구현 가능하고요. Phase 1에서 같이 넣을 수 있어요.

## 안건 3: Phase 1 태스크 분해 및 일정 수립

**PM**: 마지막 안건입니다. Phase 1 런칭까지의 태스크를 분해하고 순서와 일정을 잡겠습니다.

**FE 엔지니어**: 개발 태스크를 크게 네 단계로 나누겠습니다.

첫 번째 단계는 **프로젝트 초기 셋업**입니다. Next.js 프로젝트 생성, TypeScript와 Tailwind 설정, 프로젝트 구조 잡기, Vercel 배포 파이프라인 연결, 린트와 포맷터 설정까지요.

두 번째 단계는 **공통 컴포넌트 및 레이아웃**입니다. GNB, 푸터, 검색 기능, 계산기 공통 템플릿 컴포넌트, 광고 슬롯 컴포넌트, 결과 공유 기능을 만듭니다.

세 번째 단계는 **킬러 계산기 3종 개발**입니다. 주택대출 시뮬레이터, 연봉 실수령액 분석기, 투자 복리 계산기 순으로 가겠습니다. 각 계산기마다 계산 로직, UI, 차트 시각화, 계산기 간 연결을 구현합니다.

네 번째 단계는 **일반 계산기 1차 배치 7종 개발**입니다. 템플릿이 이미 있으니까 설정 파일과 계산 로직만 추가하면 됩니다.

**BE 엔지니어**: 여기에 병렬로 진행할 작업이 있습니다. 금리 데이터와 환율 데이터를 빌드 타임에 가져오는 데이터 fetch 스크립트를 만들어야 해요. 금융감독원 API 연동, 한국은행 기준금리 API 연동, 환율 API 연동 이 세 가지입니다. 이건 두 번째 단계와 병렬로 진행할 수 있어요.

**마케터**: SEO 작업도 병렬로 들어가야 합니다. 각 페이지의 메타 태그 템플릿, 구조화 데이터 JSON-LD 템플릿, 사이트맵 자동 생성, robots.txt 설정, 각 계산기 하단 가이드 텍스트 작성. 텍스트 작성은 제가 담당하고, 기술 적용은 FE에서 해주시면 됩니다.

**UX/UI 디자이너**: 디자인 작업 일정은 이렇습니다. 먼저 디자인 토큰을 정의해서 FE에 전달하고, 그 다음 홈페이지와 킬러 계산기 페이지 시안을 작업합니다. 일반 계산기 페이지는 킬러 계산기 디자인에서 변형하면 되니까 별도 시안이 필요 없어요.

**PM**: 일정을 잡겠습니다. 현실적으로 개발 리소스가 제한적이니까 타이트하게 가야 합니다.
