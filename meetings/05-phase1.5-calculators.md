# 5차 미팅: 건강·날짜·생활 카테고리 계산기 선정

**일시**: 2026-04-13
**참석**: PO, PM, UX/UI 디자이너, 프로덕트 디자이너, FE 엔지니어, BE 엔지니어, 마케터
**목적**: Phase 1.5 — 건강, 날짜, 생활 3개 카테고리에 추가할 계산기 선정

---

## 회의 내용

**PO**: 좋습니다, Phase 1에서 금융과 변환 카테고리는 채웠는데, 건강·날짜·생활 3개 카테고리가 아직 비어있어요. Phase 1.5로 빠르게 채울 계산기를 선정하려고 합니다. 기준은 두 가지예요. 하나는 검색 볼륨이 충분한가, 다른 하나는 구현 난이도가 낮아서 빠르게 배포할 수 있는가.

**마케터**: 제가 키워드 리서치를 좀 해봤는데요. 건강 쪽은 "BMI 계산기", "칼로리 계산기", "기초대사량 계산기"가 월 검색량 10만 이상이에요. "체지방률 계산기"도 꽤 되고요. 날짜 쪽은 "D-Day 계산기", "날짜 계산기", "만 나이 계산기"가 압도적이에요. 특히 만 나이 계산기는 2023년 만 나이 통일법 이후로 수요가 꾸준합니다. 생활 쪽은 "학점 계산기", "전기요금 계산기", "속도 단위 변환"이 상위권이에요.

**프로덕트 디자이너**: 검색량도 중요한데, 우리 사이트 특성상 계산기 간 연결이 되면 좋잖아요. 건강 카테고리에서 BMI → 기초대사량 → 칼로리 이렇게 흐름을 만들 수 있어요. "내 BMI가 이 정도면 하루에 얼마나 먹어야 하나?" 이런 시나리오요.

**UX/UI 디자이너**: 동의합니다. 그리고 건강 쪽은 시각화 요소가 매력적이에요. BMI 게이지 차트, 칼로리 파이차트 같은 거 넣으면 체류 시간 늘릴 수 있어요.

**PM**: 구현 관점에서 정리해볼게요. 건강 카테고리부터 가볼까요? 후보를 쭉 나열해보면요.

**BE 엔지니어**: 건강 쪽은 외부 API 의존이 거의 없어서 전부 클라이언트 단에서 처리 가능해요. 공식도 표준화되어 있고. BMI는 체중÷신장² 한 줄이고, 기초대사량은 해리스-베네딕트 공식이나 미플린-세인트 조어 공식 쓰면 되고, 칼로리는 기초대사량에 활동계수 곱하면 끝이에요.

**FE 엔지니어**: 맞아요, 일반 계산기 템플릿으로 하루에 2~3개씩 찍어낼 수 있는 수준이에요. 체지방률도 해군 공식 쓰면 되는데, 입력 항목이 목둘레, 허리둘레 같은 게 들어가서 사용자가 좀 귀찮아할 수 있어요.

**마케터**: 체지방률은 검색량은 있는데 이탈률이 높을 수 있어요. 측정이 번거로우니까. 차라리 심플한 것 위주로 가고, 체지방률은 나중에 추가하는 게 나을 것 같아요.

**PO**: 좋습니다. 건강은 BMI, 기초대사량, 칼로리 3개로 가죠. 시나리오 플로우도 만들 수 있고. 날짜 카테고리는요?

**프로덕트 디자이너**: D-Day 계산기는 무조건 들어가야 해요. "특정 날짜까지 며칠 남았나"는 연중 수요가 있어요. 시험, 결혼, 출산 등등. 그리고 날짜 차이 계산기도 필요해요. "두 날짜 사이 며칠인가"요.

**마케터**: 만 나이 계산기는 꼭 넣어야 해요. 키워드 경쟁도가 좀 있긴 한데, 우리가 생년월일 입력하면 만 나이뿐만 아니라 다음 생일까지 D-Day도 같이 보여주면 차별화 포인트가 됩니다.

**UX/UI 디자이너**: 날짜 카테고리에 "근무일수 계산기"는 어때요? 공휴일 제외하고 영업일 기준으로 며칠인지. 직장인들 연차 계산할 때 많이 검색하거든요.

**PM**: 공휴일 데이터가 필요하겠네요. 매년 바뀌니까.

**BE 엔지니어**: 한국천문연구원 공휴일 API가 있어요. 그런데 빌드 타임에 fetch 해서 JSON으로 넣어두면 되니까 크게 어렵진 않아요. 대체공휴일까지 반영하려면 좀 신경 써야 하지만요.

**FE 엔지니어**: 아니면 처음에는 2026년 공휴일을 하드코딩해놓고, 나중에 API 연동하는 것도 방법이에요. 빠르게 배포하려면요.

**PO**: 실용적이네요. 날짜는 D-Day, 날짜 차이, 만 나이, 근무일수 4개로 가죠.

**프로덕트 디자이너**: 생활 카테고리는 좀 잡다한 느낌이 될 수밖에 없는데, 그게 오히려 롱테일 SEO에는 유리해요.

**마케터**: 학점 계산기는 대학생 타겟으로 매학기 초·말에 검색량이 폭증해요. 전기요금 계산기는 여름·겨울 시즌성이 있고요. 그리고 "퍼센트 계산기"가 생각보다 검색량이 엄청 높아요. "100의 몇 퍼센트는?" 같은 단순한 건데, 의외로 사람들이 많이 찾아요.

**UX/UI 디자이너**: 퍼센트 계산기는 3가지 모드를 넣으면 돼요. "A의 B%는?", "A는 B의 몇 %?", "A에서 B로 변할 때 변화율은?". 입력 2개에 결과 1개니까 초심플하고.

**FE 엔지니어**: 구현 30분이면 됩니다. 학점 계산기가 좀 더 손이 가요. 학점 체계가 학교마다 다르거든요. 4.3 만점, 4.5 만점 같은 거.

**PM**: 4.3과 4.5 만점 선택할 수 있게 하면 되겠죠. 과목명, 학점, 성적 입력하는 동적 리스트 UI가 필요하긴 한데, 그렇게 복잡하진 않아요.

**BE 엔지니어**: 전기요금은 한전 누진세 구간이 있어서, 그 데이터를 정리해놔야 해요. 주거용이랑 일반용 구분도 필요하고. 근데 그것도 정적 데이터라 빌드 타임에 넣어두면 됩니다.

**마케터**: 추가로 "랜덤 뽑기" 같은 유틸도 괜찮아요. 점심 메뉴 뽑기, 번호 추첨 같은 건 소셜 공유가 많이 일어나서 바이럴 효과가 있어요.

**PO**: 재밌는 제안인데, 계산기 사이트 정체성이랑 좀 안 맞을 수 있어요. 나중에 별도 섹션으로 고려하고, 지금은 계산기에 집중하죠.

**마케터**: 맞아요, 그럼 뺄게요.

**PM**: 생활은 학점 계산기, 전기요금 계산기, 퍼센트 계산기 3개로 가면 될까요?

**프로덕트 디자이너**: 하나 더 제안하자면, "속도 단위 변환기"는 어때요? km/h ↔ m/s ↔ mph. 변환 카테고리에 넣을 수도 있지만, 생활에 넣어도 무방하고.

**PO**: 변환 카테고리가 이미 있으니까 거기 넣는 게 자연스러울 것 같은데, 생활 카테고리도 채워야 하니까... 일단 생활에 넣죠.

**UX/UI 디자이너**: 정리하면 생활은 학점, 전기요금, 퍼센트, 속도 변환 4개요?

**PO**: 네, 그렇게 가시죠. 총 11개 추가인데, 전부 일반 계산기 템플릿이니까 일정은?

**PM**: FE 엔지니어 의견으로는 일반 계산기 하나에 반나절 정도면 될 것 같은데, 맞나요?

**FE 엔지니어**: 네, 계산 로직이 단순한 것들 위주라서. 건강 3개, 날짜 4개, 생활 4개 해서 11개면 3~4일이면 충분해요. 근무일수 계산기만 공휴일 처리 때문에 좀 더 걸리겠지만요.

**PM**: 좋습니다. 그러면 일주일 안에 Phase 1.5 배포 가능하겠네요.

**PO**: 좋아요, 이걸로 확정합니다.

| **생활** | 속도 단위 변환기 | km/h ↔ m/s ↔ mph | |
