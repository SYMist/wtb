## Part 1: 아이템 방향성 논의

**PM**: 첫 미팅이니 목적부터 정리하겠습니다. 오늘은 **"애드센스 수익화에 적합한 웹 제품 아이템 선정"**이 목표입니다. 애드센스 계정은 확보된 상태이므로 승인 이슈는 넘어가고, 어떤 콘텐츠/서비스로 트래픽을 만들지 결정합시다.

**PO**: 애드센스 수익의 본질은 결국 **페이지뷰 x CTR x CPC**입니다. 이 세 가지를 동시에 높일 수 있는 아이템이어야 해요. 단순 블로그는 경쟁이 치열하고, 도구형 서비스는 체류시간이 짧아요. 저는 **검색 유입이 꾸준하면서도 반복 방문이 가능한 유틸리티+콘텐츠 혼합형**을 제안합니다.

**마케터**: 동의합니다. SEO 관점에서 롱테일 키워드를 대량으로 잡을 수 있는 구조가 핵심이에요. 제가 보기에 가능성 있는 카테고리는 세 가지입니다:

1. **개발자 도구/치트시트 사이트** — 검색량 높고 CPC도 괜찮음
2. **생활 계산기 모음 사이트** — 대출, 세금, 단위변환 등 범용적 수요
3. **텍스트/이미지 변환 도구 모음** — JSON formatter, 이미지 리사이즈 등

**UX/UI 디자이너**: 2번과 3번이 UX 측면에서 유리합니다. 사용자 의도가 명확하고 태스크 완료까지의 흐름이 짧아요. 광고 배치와 사용성을 양립시키기도 쉽고요. 1번은 콘텐츠 제작 부담이 큽니다.

**프로덕트 디자이너**: 저도 **도구 모음 사이트**가 맞다고 봅니다. 이유는 페이지 구조가 정형화되기 때문이에요. 하나의 도구 템플릿을 만들면 수십 개 페이지로 확장 가능하고, 각 페이지가 독립적인 SEO 랜딩이 됩니다. 정보 설계가 깔끔해져요.

**FE 엔지니어**: 기술적으로도 도구 모음이 합리적입니다. 대부분의 변환/계산 로직이 클라이언트 사이드에서 처리 가능하니 **정적 사이트(Next.js SSG)로 배포**하면 서버 비용이 거의 없어요. 수익 대비 운영비 구조가 좋습니다.

**BE 엔지니어**: 맞습니다. 초기에는 백엔드 없이 시작 가능하고, 나중에 API가 필요한 도구(예: QR 생성, PDF 변환)가 추가되면 그때 가벼운 서버리스 함수로 붙이면 됩니다. 인프라 부담이 최소화돼요.

**PO**: 그러면 2번과 3번을 합치는 방향은 어떻습니까? **"온라인 도구 모음 사이트"** — 생활 계산기 + 텍스트/개발 유틸리티를 하나의 사이트에서 제공하는 겁니다. 카테고리를 나누면 서로 다른 검색 유입 풀을 동시에 확보할 수 있어요.

**마케터**: 좋습니다. 다만 초기 집중이 필요합니다. 한쪽 카테고리에서 SEO 권위를 먼저 확보하고 확장하는 게 낫습니다. **생활 계산기 쪽이 타겟 유저가 훨씬 넓어요.** 대출이자 계산, 연봉 실수령액, BMI, 날짜 계산 같은 건 비개발자도 검색합니다. 트래픽 볼륨에서 압도적이에요.

**PM**: 반론 없으면 정리하겠습니다.

**UX/UI 디자이너**: 한 가지만. 사이트 이름과 정체성이 "계산기 사이트"로 고정되면 나중에 텍스트 도구 쪽으로 확장할 때 브랜딩이 어색해질 수 있어요. **포괄적인 네이밍**으로 가되, 첫 카테고리는 계산기로 시작하는 게 좋겠습니다.

**프로덕트 디자이너**: 동의합니다. "OO Tools" 같은 포괄적 브랜드 아래 계산기 카테고리부터 런칭하는 구조면 확장성 문제 없습니다.

### Part 1 중간 정리

| 항목 | 결정 사항 |
|---|---|
| **제품 아이템** | 온라인 도구 모음 웹사이트 (Online Tools) |
| **1차 카테고리** | 생활 계산기 (대출, 세금, 단위변환, 건강, 날짜 등) |
| **확장 계획** | 텍스트/개발자 유틸리티 (JSON, 인코딩, 포맷 변환 등) |
| **수익 모델** | Google AdSense (이미 계정 보유) |
| **기술 방향** | 정적 사이트(SSG) 기반, 서버 비용 최소화 |
| **브랜딩 방향** | 포괄적 네이밍으로 확장 가능한 구조 |
| **SEO 전략** | 도구별 독립 페이지로 롱테일 키워드 대량 확보 |

## Part 2: 크롤링 콘텐츠 트랙 추가 논의

**PM**: 크롤링을 통한 콘텐츠 확보 방안이 추가 안건으로 들어왔습니다. 도구 모음과 병행할 수 있는지, 어떤 형태가 적합한지 논의합시다.

**마케터**: 사실 도구 사이트만으로는 **페이지 수 확장에 한계**가 있어요. 계산기 30개 만들면 30페이지인데, 크롤링 콘텐츠를 붙이면 수백~수천 페이지를 빠르게 확보할 수 있습니다. 애드센스 수익은 결국 페이지뷰 총량 싸움이에요.

**PO**: 동의합니다. 다만 아무 콘텐츠나 긁어오면 **애드센스 정책 위반**으로 계정 정지 리스크가 있어요. 크롤링 대상은 반드시 아래 조건을 충족해야 합니다:

1. **공공데이터 또는 오픈 API** — 저작권 문제 없음
2. **원본 데이터에 가공/분석을 더함** — 단순 복제가 아닌 부가가치 창출
3. **사용자에게 실질적 유용성** — 스팸성 페이지가 아닌 것

**BE 엔지니어**: 기술적으로 괜찮은 소스가 몇 가지 있습니다:

- **공공데이터포털 (data.go.kr)** — 부동산 실거래가, 약국/병원 위치, 물가 정보
- **기상청 API** — 날씨 데이터 기반 지역별 페이지
- **국가통계포털 (KOSIS)** — 인구, 경제 지표
- **환율/주가 API** — 실시간 금융 정보

이런 데이터를 주기적으로 크롤링해서 정적 페이지로 빌드하면 됩니다.

**프로덕트 디자이너**: 이걸 도구 사이트와 **자연스럽게 연결**할 수 있어요. 예를 들어:

- 부동산 실거래가 데이터 → **대출이자 계산기**로 연결
- 물가 정보 → **생활비 계산기**로 연결
- 환율 데이터 → **환율 변환 도구**로 연결

콘텐츠가 도구로의 내부 링크 역할을 하면 SEO 시너지가 납니다.

**UX/UI 디자이너**: 좋은 구조인데, 주의할 점이 있어요. 크롤링 콘텐츠 페이지가 **데이터 나열에 그치면 체류시간이 극도로 짧아집니다.** 차트, 비교 테이블, 트렌드 시각화 같은 **가공된 뷰**를 반드시 제공해야 해요. 그래야 광고 노출 기회도 생기고요.

**FE 엔지니어**: 빌드 타임에 데이터를 fetch해서 정적 페이지로 만들면 됩니다. Next.js의 ISR(Incremental Static Regeneration)을 쓰면 데이터 갱신 주기도 조절 가능하고, 서버 비용은 여전히 최소로 유지됩니다.

**마케터**: 크롤링 콘텐츠 후보를 SEO 관점에서 우선순위를 매기면:

| 순위 | 콘텐츠 | 이유 |
|---|---|---|
| 1 | **부동산 실거래가/시세** | 검색 볼륨 매우 높음, 지역별 페이지로 수천 페이지 생성 가능 |
| 2 | **환율/금 시세** | 매일 검색 발생, 반복 방문 유도 |
| 3 | **공공 생활정보** (약국, 병원, 주차장) | 지역 기반 롱테일 키워드 대량 확보 |
| 4 | **물가/경제 지표** | CPC 높은 금융 키워드와 연결 |

**PO**: 1번과 2번은 이미 강력한 경쟁자(호갱노노, 네이버 부동산 등)가 있어요. 차별화 없이 들어가면 매몰됩니다.

**마케터**: 맞아요. 그래서 핵심은 **데이터 + 도구의 결합**입니다. 경쟁 사이트들은 데이터만 보여주는데, 우리는 데이터를 보고 바로 계산기로 시뮬레이션할 수 있게 하는 거예요. "이 아파트 시세 보기 → 대출 시뮬레이션" 이 흐름이 차별점이 됩니다.

**PM**: 다만 초기 리소스를 고려하면 두 트랙을 동시에 풀파워로 진행하긴 어렵습니다. 우선순위를 정합시다.

**PO**: **트랙 A(계산기 도구)를 먼저 런칭하고, 트랙 B(크롤링 콘텐츠)를 2차로 붙이는 게** 맞다고 봅니다. 도구는 한번 만들면 유지보수가 적고, 크롤링은 데이터 파이프라인 구축에 초기 공수가 더 들어요.

**BE 엔지니어**: 동의합니다. 크롤링 파이프라인은 안정화에 시간이 필요합니다. 도구 먼저 올리고 그 사이에 파이프라인을 준비하는 게 효율적이에요.

## 최종 결론

| 항목 | 결정 사항 |
|---|---|
| **제품 구조** | 도구 + 크롤링 콘텐츠 결합형 웹사이트 |
| **Phase 1** | 생활 계산기 도구 모음 런칭 |

| **핵심 차별점** | 데이터 열람 → 도구 시뮬레이션 연결 (콘텐츠-도구 시너지) |
| **크롤링 소스 후보** | 공공데이터포털, 환율 API, 기상청, KOSIS 등 |
| **콘텐츠 원칙** | 단순 나열 금지, 가공/시각화 필수, 저작권 안전한 소스만 사용 |
| **수익 모델** | Google AdSense |

| **SEO 전략** | 도구별/콘텐츠별 독립 페이지로 롱테일 키워드 대량 확보 |
